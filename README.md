# Namshi Campaign Generation Engine — Layer 1

Generates a draft monthly campaign calendar from 15 months of historical data + a regional event config.
The output is always a human-editable draft. Nothing is auto-published.

---

## Quick start

```bash
# 1. Install dependencies (Python 3.10+)
pip install -r requirements.txt

# 2. Standardize history (run once, re-run if the xlsx changes)
python src/standardize.py

# 3. Generate a draft calendar
python src/generate.py --month 7 --year 2026
```

Outputs land in `output/`:
- `draft_calendar_7_2026.xlsx` — the draft in Standard Schema
- `draft_notes.md` — assumptions, flagged rows, gaps

## The five steps

| Step | Script | What it does |
|---|---|---|
| 1 | `standardize.py` | Reads all sheets → one tidy `standardized_history.csv` |
| 2 | `recurrence.py` | Groups by canonical key → knowledge base (typical tier, geo, dates, duration) |
| 3–5 | `generate.py` | Resolves target month's events from config + KB → draft xlsx + notes |

---

## Config ownership

`config/calendar_config.yaml` is the specialist's file. Key things to maintain:

### Before every planning cycle
- **Lunar dates** (`lunar_events` → `dates`): Ramadan, Eid al-Fitr, Eid al-Adha, Diwali, Holi shift every year. Enter confirmed dates from the Hijri calendar or official announcements.
- **Sports fixtures** (`sports_events` → `dates`): World Cup, F1, marathons, Dubai Active. Enter from official fixture lists.
- **Dubai Shopping Festival**: announced annually by Dubai. Update `needs_date_confirmation: true` entries.

### Controlled vocabularies (do not change keys)
- `geography_normalization` — maps raw variants → `KSA + UAE` / `KSA` / `UAE` / `All`
- `tier_normalization` — maps raw variants → `Tier 1` / `Tier 2` / `Tier 3` / `-`
- `campaign_name_map` — maps name variants from history → canonical keys

## `needs_review` logic

A generated row is flagged `TRUE` when any of:
- Config has `needs_date_confirmation: true` for that event
- Dates were estimated from history (offset +1 year)
- No historical precedent exists (`source = "config"`)
- It's a trend edit suggestion
- Beauty Week (active-month confirmation required)

Rows with `needs_review = FALSE` are high-confidence: Payday (rule: 25th), Feed Refresh (rule: ~7th), and events confirmed in the current year's history (e.g. EOSS Jun 2026 from the June 2026 sheet).

---

## Date estimation priority

For each event `generate.py` tries in order:

1. **Config explicit dates** — lunar / sports events with `dates.YYYY.start/end`
2. **Hard-coded rules** — Payday (~25th), Feed (~7th), Father's Day (3rd Sunday June)
3. **History: same-year or offset** — most_recent_start/end from KB, offset +1yr if from prior year
4. **History: typical start day** — only if the campaign typically starts in this month
5. **Seasonal window fallback** — full target month (marks `needs_review = TRUE`)

## Known data-quality notes

- **`april 2025` sheet** is excluded — its schema is wireframe-level (Theme/Merch/Country) and cannot be safely mapped to the standard schema.
- **`AE + GCC`** geography is normalised to `UAE`.
- **November 2025** has a split promo schema (5 promo columns) — these are concatenated into `promos_coupons` with `|` separators.
- **Complex tier values** (e.g. "Tier 1 on Men / Tier 2 Women") are reduced to the highest tier present; original is preserved in `raw_notes`.
- **Serial dates** are converted via epoch `1899-12-30`. Note: the brief's spot-check example `46178 → 2026-06-08` contains a typo — the correct mapping is `46178 → 2026-06-05` and `2026-06-08 → serial 46181`. The conversion formula is correct.
